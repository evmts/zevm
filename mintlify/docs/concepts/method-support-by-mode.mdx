# Method Support By Mode

ZEVM's public contract separates canonical surfaces, compatibility aliases, and deferred surfaces by mode.

For exact request tuples, return payloads, selector rules, and error behavior, use the public reference pages: [Core Reads](../reference/json-rpc/core-reads), [Simulation](../reference/json-rpc/simulation), [Transactions And Mining](../reference/json-rpc/transactions-and-mining), [Blocks, Receipts, And Logs](../reference/json-rpc/blocks-receipts-and-logs), [ZEVM Controls](../reference/json-rpc/dev-controls), [Verified Light-Mode Reads](../reference/json-rpc/verified-light-mode-reads), and [Unsupported And Deferred](../reference/json-rpc/unsupported-and-deferred).

## Canonical Surfaces

| Surface | Trusted mode | Light mode | Canonical public naming |
| --- | --- | --- | --- |
| Core reads, including `eth_chainId` and `eth_blockNumber` | yes | yes | standard `eth_*` reads |
| Simulation | yes | no | `eth_call`, `eth_estimateGas` |
| Transaction submission and mining | yes | no | transaction `eth_*` methods plus `zevm_*` mining controls |
| Log queries, including `eth_getLogs` | yes | no | canonical block, receipt, and log queries |
| Snapshots and revert | yes | no | canonical `zevm_*` snapshot controls |
| State mutation, impersonation, and time controls | yes | no | canonical `zevm_*` dev-node controls |
| Fork overlay configuration | yes | no | trusted-mode config plus canonical `zevm_*` controls |
| Verified light-mode status | no | yes | `zevm_lightSyncStatus` |
| Proof-backed reads | no | yes | standard `eth_*` reads only |
| Filter lifecycle APIs beyond `eth_getLogs` | no | no | deferred |
| Subscriptions | no | no | deferred |
| WebSocket transport | no | no | deferred |

For light-mode proof-backed reads, selector support is also mode-specific:

- `latest`, `safe`, `finalized`, and `earliest` are part of the intended light-mode contract
- while `ready = false`, `eth_blockNumber` fails with `-32011`
- once ready, `eth_blockNumber` returns the block number of the light-mode `latest` head
- while `ready = false`, ZEVM serves no proof-backed reads and fails them with `-32011`
- once ready, numeric selectors are supported only for block `0` and for exact blocks inside the retained verified-history window containing the most recent `8191` verified execution blocks when ZEVM can verify the exact execution block and the requested proof-backed read against that block's state root
- if light mode is ready in general but the requested numeric block is outside that retained verified-history window, the request fails with `-32602`
- if an otherwise-supported proof-backed read cannot be verified against the resolved block state root, the read fails with the proof-verification error documented on [Verified Light-Mode Reads](../reference/json-rpc/verified-light-mode-reads)
- ZEVM does not promise arbitrary checkpoint-to-head historical archive reads

## Compatibility Aliases

Trusted mode also accepts a defined compatibility-alias set for the same trusted-mode behavior:

- the accepted `anvil_*`, `hardhat_*`, and `evm_*` names are fixed rather than open-ended
- no other same-suffix `anvil_*`, `hardhat_*`, or `evm_*` names are part of the phase-1 public contract

These aliases are supported in trusted mode only. Public docs use the canonical `zevm_*` names when describing ZEVM behavior.

## Deferred Surfaces

Trace and verbose logging tooling is phase-3 work, not phase-1 product surface. If it appears in the codebase, it should be documented as deferred rather than canonical.

Additional deferred public surfaces:

- filter lifecycle APIs beyond `eth_getLogs`
- subscriptions
- WebSocket transport

## Not Public ZEVM

- light mode does not expose trusted-mode dev controls
- trusted mode does not expose `zevm_lightSyncStatus`

## Current HEAD As Of 2026-03-29

- `zig build` fails first in `src/rpc/server.zig` because current upstream `jsonrpc` no longer exposes `jsonrpc.envelope`, and `zig build test` fails in `src/rpc/dispatcher_test.zig`, `src/rpc/server.zig`, and `src/tx_processor.zig` because current upstream `jsonrpc` no longer exposes `jsonrpc.envelope` and current `guillotine-mini` requires a wider `evm.init` signature
- `src/main.zig` still constructs an empty `dispatcher.HandlerRegistry`, so the live listener does not yet expose the modeled mode split
- the deferred bucket still excludes `eth_getLogs`, but current helper presence does not imply contract-complete trusted canonical queries: current `HEAD` still carries live `C-007` contradictions, including `eth_getTransactionByHash` remaining a stub, block/receipt/log payloads remaining placeholder-backed, and invalid log filters collapsing to `[]`
- trusted core read helpers remain contradictory too: `eth_getCode` still returns `0x`, malformed storage slots still collapse to zero, and `eth_feeHistory` still synthesizes history instead of reflecting mined blocks
- there is still no live `zevm_lightSyncStatus` route in the executable path
- there is still no live retained-history numeric-selector implementation for light mode
- light-mode `eth_blockNumber` semantics are settled in the docs contract, but no live light-mode route exists yet
- the public contract on this page is the intended mode split, not the current implementation shape
