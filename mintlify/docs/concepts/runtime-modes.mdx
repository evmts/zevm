# Runtime Modes

ZEVM has exactly two runtime modes. Startup defaults to trusted mode unless startup inputs explicitly select light mode. CLI `--mode light` selects light mode directly, and a config file whose sole runtime branch is `mode.light` does the same. Trusted mode is the writable local dev node, and light mode is the consensus-backed verified-read runtime. Forking is configuration inside trusted mode, not a third mode.

## Intended Behavior

Public docs use these surface rules:

- standard `eth_*` core reads span both modes
- trusted mode also owns simulation, transaction submission, and canonical block, receipt, and log queries, including `eth_getLogs`
- trusted mode's canonical nonstandard controls use the `zevm_*` namespace
- trusted mode also accepts a fixed compatibility-alias set under `anvil_*`, a limited `hardhat_*` subset, and selected `evm_*` names
- light mode exposes verified reads and `zevm_lightSyncStatus`, not local dev-node controls, simulation, transaction submission, or canonical block, receipt, and log queries
- filter lifecycle APIs beyond `eth_getLogs` stay deferred
- `zevm_*` is the public canonical naming for ZEVM-owned trusted-mode controls, not a docs shorthand

Exact method tuples, payloads, selectors, and accepted alias inventories are published on the public JSON-RPC reference pages: [Core Reads](../reference/json-rpc/core-reads), [Simulation](../reference/json-rpc/simulation), [Transactions And Mining](../reference/json-rpc/transactions-and-mining), [Blocks, Receipts, And Logs](../reference/json-rpc/blocks-receipts-and-logs), [ZEVM Controls](../reference/json-rpc/dev-controls), [Verified Light-Mode Reads](../reference/json-rpc/verified-light-mode-reads), and [Unsupported And Deferred](../reference/json-rpc/unsupported-and-deferred).

## Mode Boundary

| Surface | Trusted mode | Light mode |
| --- | --- | --- |
| Core reads such as `eth_chainId` and `eth_blockNumber` | yes | yes |
| Account, code, storage, and nonce reads | yes, against trusted local state | yes, when backed by verified proofs |
| `eth_call` and `eth_estimateGas` | yes | no |
| Transaction submission and mining | yes | no |
| Canonical block, receipt, and log queries, including `eth_getLogs` | yes | no |
| `zevm_*` dev-node controls | yes | no |
| the trusted-mode compatibility aliases under `anvil_*`, `hardhat_*`, and selected `evm_*` names | yes, trusted mode only | no |
| `zevm_lightSyncStatus` | no | yes |

Trusted-mode compatibility aliases do not change the public contract. They only provide alternate method names for the same trusted-mode behavior. Deferred filter lifecycle APIs are separate from canonical log queries such as `eth_getLogs`.
In light mode, `eth_blockNumber` shares the readiness gate: while `ready = false`, it fails with `-32011`; once ready, it returns the block number of the light-mode `latest` head, meaning the latest verified optimistic execution head.

## Block Tags

| Tag | Trusted mode | Light mode |
| --- | --- | --- |
| `latest` | current local head | latest verified optimistic execution head |
| `pending` | compatibility alias of `latest` | unsupported |
| `safe` | compatibility alias of `latest` | consensus-backed safe head |
| `finalized` | compatibility alias of `latest` | consensus-finalized head |
| `earliest` | block `0` | block `0` |
| numeric quantity | exact local block number | block `0`, or an exact block inside the retained verified-history window containing the most recent `8191` verified execution blocks when ZEVM can verify that exact execution block and the requested proof-backed read against that block's state root |

Trusted-mode `pending`, `safe`, and `finalized` are aliases only. They do not imply finality. In light mode, `safe` and `finalized` are real consensus-backed heads rather than aliases.

ZEVM does not promise arbitrary checkpoint-to-head historical archive reads. `ready = true` only when `status = "synced"` and ZEVM can serve proof-backed reads. While `ready = false`, ZEVM serves no proof-backed reads and fails them with `-32011`. Once ready, numeric selectors are supported only for block `0` and for exact blocks inside the retained verified-history window containing the most recent `8191` verified execution blocks when ZEVM can verify the exact execution block and the requested proof-backed read against that block's state root. If the requested numeric block is outside that retained verified-history window, the request fails with `-32602`. Verification failures for otherwise-supported proof-backed reads follow the proof-verification error contract on [Verified Light-Mode Reads](../reference/json-rpc/verified-light-mode-reads).

## Trusted-Mode Config

The trusted-mode config shape is explicit:

```json
{
  "mode": {
    "trusted": {
      "chainId": 31337,
      "mining": { "type": "auto" },
      "fork": null
    }
  }
}
```

The resolved unions are:

- `mode.trusted.mining`: `{ "type": "auto" }`, `{ "type": "manual" }`, or `{ "type": "interval", "blockTime": <seconds> }`
- `mode.trusted.fork`: `null`, `{ "url": "https://..." }`, or `{ "url": "https://...", "blockNumber": <u64> }`

- the public contract on this page is the intended product shape, not the current runtime wiring
