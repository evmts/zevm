# State Fork And Snapshots

Forking is configuration inside trusted mode. It is not a third ZEVM runtime mode.

## Intended Behavior

Trusted mode owns the writable overlay. A forked trusted node reads upstream state from the configured fork source, then applies local writes on top.

Public naming rules stay consistent here too:

- canonical snapshot, revert, and state-control methods use `zevm_*`
- a fixed compatibility-alias set under `anvil_*`, a limited `hardhat_*` subset, and selected legacy `evm_*` names is accepted in trusted mode only
- `zevm_*` is the public canonical naming for ZEVM-owned trusted-mode controls, not a docs convention

Fork config is explicit:

```json
{
  "mode": {
    "trusted": {
      "chainId": 31337,
      "mining": { "type": "manual" },
      "fork": {
        "url": "https://rpc.example",
        "blockNumber": 22000000
      }
    }
  }
}
```

The resolved fork union is:

- `null`
- `{ "url": "https://..." }`
- `{ "url": "https://...", "blockNumber": <u64> }`

`chainId` remains a trusted-mode startup setting and is not inferred from the fork source.

## Snapshot Boundary

Snapshots and revert capture the trusted-mode overlay and ZEVM-owned runtime metadata, not the upstream source. The snapshot scope is the local trusted node state, not the remote fork provider or any light-mode checkpoint state. The documented boundary includes:

- local journal or overlay state
- canonical local head
- receipt and log indexes derived from local blocks
- pending txpool state
- mining configuration
- block-environment overrides
- impersonation state
- time controls

Snapshots do not capture:

- the remote fork source itself
- light-mode checkpoint or consensus state

The canonical public controls for this boundary are `zevm_snapshot` and `zevm_revert`.

## Local State Import And Export

Local state import and export stay separate from both snapshot/revert and fork-source selection:

- `zevm_dumpState`
- `zevm_loadState`

These methods serialize or hydrate trusted-mode local state. They are not snapshot IDs, and they do not redefine which upstream fork source ZEVM reads from.

## Fork-Source Controls

Fork-source controls are separate again:

- `zevm_reset`
- `zevm_setRpcUrl`

`zevm_reset` resets trusted-mode state back to a fresh in-memory node or to a supplied fork configuration. It is a fork-source and runtime-reset control, not a snapshot restore primitive and not a dump/load import hook.

`zevm_setRpcUrl` changes the upstream RPC endpoint used for forked reads. It adjusts fork-source configuration, not the local overlay itself and not the snapshot boundary.
