# Architecture And Upstream Ownership

ZEVM is an integration shell, not a second owner of upstream execution infrastructure.

## Ownership Boundary

- Voltaire owns Ethereum primitives, JSON-RPC types, state manager, journal and snapshot support, fork backend, blockchain, crypto, and execution-layer foundations.
- `guillotine-mini` owns the EVM interpreter, tracing substrate, and execution host integration.
- ZEVM owns CLI and config parsing, mode selection, runtime composition, lifecycle, canonical HTTP JSON-RPC transport, mode-aware routing, startup and shutdown, checkpoint wiring, readiness reporting, and the public naming of nonstandard trusted-mode controls under the canonical `zevm_*` namespace.
- The canonical transport owner on the shipping path is ZEVM's `server.zig` plus `dispatcher.zig` path, not the stale local helper stack.

## Observed `HEAD` Constraints, 2026-03-29

- `zig build` fails first in `src/rpc/server.zig` because current upstream `jsonrpc` no longer exposes `jsonrpc.envelope`, and `zig build test` fails in `src/rpc/dispatcher_test.zig`, `src/rpc/server.zig`, and `src/tx_processor.zig` because current upstream `jsonrpc` no longer exposes `jsonrpc.envelope` and current `guillotine-mini` requires a wider `evm.init` signature.
- Current `HEAD` still carries duplicate-stack drift: a local `envelope` / `router` / handler stack exists alongside the upstream-backed path.
- That local stack is non-shipping drift until it is reconciled with the canonical transport path or removed.
- The build uses the sibling workspace path `../guillotine-mini`.
- Public docs intentionally separate the canonical `zevm_*` surface from compatibility aliases such as `anvil_*`, `hardhat_*`, and `evm_*` where those aliases are supported.
- These are observed implementation constraints, not a change in ownership.
