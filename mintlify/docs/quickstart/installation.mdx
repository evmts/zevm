# Installation

ZEVM is installed from source.

If you only need a local ZEVM build, complete this page through [Build ZEVM](#build-zevm) and skip the optional release-metadata path below.

## Prerequisites

- Zig `0.15.2` or newer for local source builds (phase-1 minimum toolchain: [`docs/specs/prd.md` section 3.3](https://github.com/evmts/zevm/blob/main/docs/specs/prd.md#33-phase-1-source-build-installation-contract))
- exact Zig version pinned in `release-tuple.json` when making canonical release reproducibility claims (see [Release Metadata Runbook](../reference/release-metadata-runbook))
- Git
- Local sibling checkouts of `voltaire` and `guillotine-mini` (ZEVM uses path dependencies)
- Optional tooling used by commands on other docs pages: `jq` and one artifact fetch tool (`gh` or `curl`)

## Workspace Layout

ZEVM expects this directory shape by default:

```text
<workspace>/
  zevm/
  voltaire/
  guillotine-mini/
```

Phase-1 quickstart contract assumes this sibling layout. Alternate workspace layouts and dependency-path rewrites are out of scope for this public flow.

## Clone And Set Up Sibling Repos

From a shared parent directory, clone all required repos as siblings:

```bash
mkdir -p ~/code/zevm-workspace
cd ~/code/zevm-workspace

git clone git@github.com:evmts/zevm.git zevm
git clone git@github.com:evmts/voltaire.git voltaire
git clone git@github.com:evmts/guillotine-mini.git guillotine-mini
```

If you use HTTPS, replace the clone URLs accordingly.

For canonical release-metadata reproducibility, use [Release Metadata Runbook](../reference/release-metadata-runbook).
That runbook is the authoritative phase-1 flow for selecting `releaseIdentifier`, validating `release-tuple.json` and `light-default-checkpoints.json`, handling metadata-invalid identifiers, and recording operator provenance.
Its default artifact output path (`.ops/release-metadata/$RELEASE_IDENTIFIER/`) is reused by [Run Light Mode](./run-light-mode) when selecting startup checkpoints from release metadata defaults.

## Optional: Pin Build Inputs From Release Tuple

If you want a pinned source build without running the full runbook automation:

1. Open the selected release on [ZEVM release metadata](https://github.com/evmts/zevm/releases) and fetch both `release-tuple.json` and `light-default-checkpoints.json` for the same `releaseIdentifier`.
2. Check out `zevm`, `voltaire`, and `guillotine-mini` to the pinned revisions from that tuple, then confirm each with `git rev-parse HEAD`.
3. Confirm local Zig matches tuple `zigVersion` with `zig version`.
4. Keep `light-default-checkpoints.json` available for [Run Light Mode](./run-light-mode#2-choose-startup-checkpoint-input) checkpoint selection.

For canonical public reproducibility claims, do not stop at tuple pinning alone; complete the full [Release Metadata Runbook](../reference/release-metadata-runbook).

## Build ZEVM

From the `zevm` repository root:

```bash
zig build
```

The executable is installed to:

```text
./zig-out/bin/zevm
```

## Run ZEVM

Run the installed binary directly:

```bash
./zig-out/bin/zevm
```

Or run through Zig's `run` step and pass runtime flags after `--`:

```bash
zig build run -- --mode trusted --host 127.0.0.1 --port 8545
```

## Run Tests

```bash
zig build test
```

## Setup Notes

- In this docs site, **phase 1** means the current public ZEVM contract: two runtime modes (`trusted`, `light`) and HTTP JSON-RPC only (no built-in TLS/auth).
- Missing `../voltaire` or `../guillotine-mini` causes dependency resolution failures during build.
- Align revisions across `zevm`, `voltaire`, and `guillotine-mini` before building.
- For canonical release-metadata reproducibility, follow [Release Metadata Runbook](../reference/release-metadata-runbook) end-to-end.
- If the selected `releaseIdentifier` does not publish valid `release-tuple.json` and `light-default-checkpoints.json`, do not treat that build boundary as metadata-backed; continue only with operator-recorded provenance for that source state.
- Phase-1 limitation: ZEVM does not expose `releaseIdentifier` via runtime JSON-RPC or CLI; keep operator provenance records for release/build identity.
- ZEVM phase 1 serves JSON-RPC over plain HTTP only and does not provide built-in TLS or JSON-RPC authentication.
- Default RPC listener is `127.0.0.1:8545`; binding `--host` beyond localhost exposes RPC to reachable networks.
- Keep `--host 127.0.0.1` unless remote access is required; if remote access is required, restrict ingress and place TLS/auth in front of ZEVM (reverse proxy or gateway).
