# Forked Dev Node

Forked trusted mode is trusted mode with `mode.trusted.fork` set, or the equivalent `--fork-url` CLI flag.

- `zig build test` currently fails in `src/rpc/dispatcher_test.zig`, `src/rpc/server.zig`, and `src/tx_processor.zig` because current upstream `jsonrpc` no longer exposes `jsonrpc.envelope` and current `guillotine-mini` requires a wider `evm.init` signature

- config loading is not yet part of the startup flow

## Intended Behavior

Forking stays inside trusted mode and does not create a third product mode. The fork configuration mirrors the CLI surface:

- `fork.url` mirrors `--fork-url`
- `fork.blockNumber` mirrors `--fork-block-number`
- omitting `fork.blockNumber` means fork from the upstream latest block
- `fork` does not implicitly change `chainId`
- `--fork-block-number` without `--fork-url` is a hard startup failure

## Intended Startup

Forked trusted mode starts with the same binary as the default trusted node:

```bash
./zig-out/bin/zevm --mode trusted --fork-url https://rpc.example --fork-block-number 22000000 --mining manual
```

If a config file already selects `mode.trusted`, `--mode trusted` is optional and only confirms the branch.

If you want a time-based local chain on top of the fork, use interval mining:

```bash
./zig-out/bin/zevm --mode trusted --fork-url https://rpc.example --mining interval --block-time 12
```

## Local Overlay Semantics

Forked trusted mode keeps ZEVM's local dev-node behavior on top of remote state:

- reads resolve from the local trusted overlay first
- if the local overlay does not provide the requested state, ZEVM falls back to the upstream fork source
- all writes, mining, snapshots, impersonation, and time controls remain local to ZEVM and do not mutate the upstream RPC
- `chainId` stays a trusted-mode startup setting and is not inferred from `fork.url`

This means a forked dev node is still a local trusted runtime. The fork is a read backing source, not a remote execution owner.
