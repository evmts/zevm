# ZEVM Maintainer Decisions

Last updated: 2026-03-29

These are support-layer control decisions. They refine exact detail and record explicit repo-local stances for settled surfaces, but they do not silently outrank `docs/specs/prd.md` or `docs/specs/docs-first-process.md`. When a decision carries a temporary public-doc stance for an unresolved implementation contradiction, the related question trail must stay auditable in `docs/specs/open-questions.md`.

| Decision ID | Topic | Status | Current contract stance | Related question |
| --- | --- | --- | --- | --- |
| `DEC-001` | docs are full-spec artifacts, not conceptual sketches | `resolved` | public and internal docs must specify exact flags, config fields, defaults, precedence, invalid combinations, method names, and error behavior | `-` |
| `DEC-002` | one binary, two modes, and forking stays inside trusted mode | `resolved` | ZEVM ships one `zevm` binary with `trusted` and `light`; forking is trusted-mode configuration, not a third mode | `-` |
| `DEC-003` | canonical transport owner | `resolved` | the canonical shipping transport owner is `src/rpc/server.zig` plus `src/rpc/dispatcher.zig` for HTTP JSON-RPC 2.0 rooted at `/`; the in-tree `src/rpc/envelope.zig` plus `src/rpc/router.zig` stack is stale or non-shipping while it remains in the tree | `-` |
| `DEC-004` | notification semantics | `resolved` | missing-`id` notifications produce no JSON-RPC response; `"id": null` is not a notification | `-` |
| `DEC-005` | trusted-mode block tags | `resolved` | in trusted mode, `pending`, `safe`, and `finalized` are compatibility aliases of `latest` | `-` |
| `DEC-006` | exact managed dev-wallet contract | `resolved` | the mnemonic, derivation path, address order, and private-key table in the PRD are public and exact | `-` |
| `DEC-007` | light-mode checkpoint precedence | `resolved` | explicit checkpoint wins over persisted checkpoint, which wins over baked network defaults | `-` |
| `DEC-008` | light-mode checkpoint file contract | `resolved` | `checkpointDir/checkpoint` stores exactly 64 hex characters without `0x`; whitespace is ignored on load | `-` |
| `DEC-009` | light-mode readiness surface | `resolved` | `zevm_lightSyncStatus` is part of the intended public contract; `ready = true` only when `status = "synced"` and ZEVM can serve proof-backed reads, and while `ready = false` proof-backed read methods fail with the light-mode not-ready error | `-` |
| `DEC-010` | deferred surfaces | `resolved` | tracing, filter lifecycle APIs beyond `eth_getLogs`, subscriptions, and WebSocket transport stay deferred until after canonical trusted and light surfaces; `eth_getLogs` remains part of the supported trusted-mode canonical query surface and is not deferred | `Q-001` |
| `DEC-011` | installation distribution promise | `resolved` | phase 1 promises source-build installation only; packaged binaries are not part of the published phase-1 contract | `Q-002` |
| `DEC-012` | exact nonstandard trusted-mode method inventory and alias policy | `resolved` | ZEVM-owned canonical nonstandard methods use the `zevm_*` namespace; accepted Anvil-prefixed aliases are limited to `anvil_addBalance`, `anvil_autoImpersonateAccount`, `anvil_deal`, `anvil_dealErc20`, `anvil_dropAllTransactions`, `anvil_dropTransaction`, `anvil_dumpState`, `anvil_getAutomine`, `anvil_getIntervalMining`, `anvil_impersonateAccount`, `anvil_increaseTime`, `anvil_loadState`, `anvil_metadata`, `anvil_mine`, `anvil_mineDetailed`, `anvil_nodeInfo`, `anvil_removeBlockTimestampInterval`, `anvil_removePoolTransactions`, `anvil_reset`, `anvil_revert`, `anvil_setAutomine`, `anvil_setBalance`, `anvil_setBlockGasLimit`, `anvil_setBlockTimestampInterval`, `anvil_setChainId`, `anvil_setCode`, `anvil_setCoinbase`, `anvil_setErc20Allowance`, `anvil_setIntervalMining`, `anvil_setMinGasPrice`, `anvil_setNextBlockBaseFeePerGas`, `anvil_setNextBlockTimestamp`, `anvil_setNonce`, `anvil_setRpcUrl`, `anvil_setStorageAt`, `anvil_setTime`, `anvil_snapshot`, and `anvil_stopImpersonatingAccount`; accepted Hardhat-prefixed aliases are limited to `hardhat_dropTransaction`, `hardhat_getAutomine`, `hardhat_impersonateAccount`, `hardhat_metadata`, `hardhat_mine`, `hardhat_reset`, `hardhat_setBalance`, `hardhat_setCode`, `hardhat_setCoinbase`, `hardhat_setMinGasPrice`, `hardhat_setNextBlockBaseFeePerGas`, `hardhat_setNonce`, `hardhat_setStorageAt`, and `hardhat_stopImpersonatingAccount`; accepted shared `evm_*` aliases are limited to `evm_mine`, `evm_snapshot`, `evm_revert`, `evm_increaseTime`, `evm_setAutomine`, `evm_setIntervalMining`, `evm_setNextBlockTimestamp`, and `evm_setBlockGasLimit`; deferred advanced aliases are limited to `anvil_enableTraces`, `hardhat_addCompilationResult`, `hardhat_setLoggingEnabled`, and `hardhat_setPrevRandao` | `-` |
| `DEC-013` | exact trusted-mode config JSON subshapes for non-auto mining and non-null fork config | `resolved` | `mode.trusted.mining` allows `{ \"type\": \"auto\" }`, `{ \"type\": \"manual\" }`, or `{ \"type\": \"interval\", \"blockTime\": <seconds> }`; `mode.trusted.fork` allows `null`, `{ \"url\": \"https://...\" }`, or `{ \"url\": \"https://...\", \"blockNumber\": <u64> }`; `chainId` remains top-level under `mode.trusted` | `-` |
| `DEC-014` | control-doc usage and explicit divergence rule | `resolved` | `docs/specs/prd.md` remains the authoritative intended phase-1 product definition and `docs/specs/docs-first-process.md` remains the process constraint; `docs/specs/json-rpc-contract.md`, `docs/specs/maintainer-decisions.md`, and `docs/specs/internal/*` may backfill exact detail or record an explicit contradiction plus public-doc stance for a surface, but they do not silently override the higher-order docs | `-` |
| `DEC-015` | compatibility aliases share ZEVM semantics | `resolved` | accepted `anvil_*`, `hardhat_*`, and `evm_*` aliases are alternate method names for the same canonical `zevm_*` contract and therefore share ZEVM params, return payloads, mode gating, and error behavior exactly | `-` |
| `DEC-016` | light-mode checkpoint startup failures vs reserved RPC codes | `resolved` | if the initially selected light-mode checkpoint is stale under strict age policy or malformed or corrupt, ZEVM fails startup before opening the listener; `-32012` and `-32013` remain reserved ZEVM light-mode codes and are not part of the initial proof-backed read method surface in this pass | `-` |
| `DEC-017` | authoritative trusted-mode runtime and config model | `resolved` | docs describe one authoritative top-level trusted-mode runtime and config model owned by startup and mode selection; `DevRuntime` may remain an implementation detail or prototype helper only and must not be documented as a competing public runtime model | `-` |
| `DEC-018` | light-mode checkpoint age boundary and non-strict branch | `resolved` | a checkpoint is stale only when `age > maxCheckpointAgeSeconds`; `age == maxCheckpointAgeSeconds` is valid; when stale and `strictCheckpointAge = false`, ZEVM warns and continues, and when stale and `strictCheckpointAge = true`, ZEVM fails startup before listening | `-` |
| `DEC-019` | light-mode numeric block-selector contract | `resolved` | while `ready = false`, ZEVM serves no proof-backed reads and fails them with `-32011`; once ready, numeric selectors are supported only for block `0` and for exact blocks inside the retained verified-history window containing the most recent `8191` verified execution blocks when ZEVM can verify the exact execution block and the requested proof-backed read against that block's state root; if light mode is ready in general but the requested numeric block is outside that retained verified-history window, the request fails with `-32602`; `earliest` remains block `0`; ZEVM does not promise arbitrary checkpoint-to-head historical archive reads | `Q-003` |
| `DEC-020` | baked checkpoint defaults are precedence inputs, not frozen public hashes | `resolved` | the public contract guarantees that baked network defaults exist and participate in checkpoint precedence, but the exact literal default checkpoint hashes remain implementation defaults rather than frozen public API or compatibility guarantees | `-` |
| `DEC-021` | light-mode `eth_blockNumber` readiness and head meaning | `resolved` | in light mode, `eth_blockNumber` fails with `-32011` while `ready = false`; once `ready = true`, it returns the block number of the light-mode `latest` head, meaning the latest verified optimistic execution head | `Q-004` |
| `DEC-022` | exact `--config` loader failure behavior | `resolved` | if `--config` points to a missing file, an unreadable file, or malformed JSON, ZEVM fails startup before opening the listener, exits non-zero, reports an operator-facing error that names the path and failure class, and does not fall back to defaults | `Q-005` |
| `DEC-023` | exact empty-batch `[]` transport behavior | `resolved` | HTTP `POST /` with body `[]` returns HTTP `200` and a single JSON-RPC error object with `jsonrpc: "2.0"`, `id: null`, and `error: { "code": -32600, "message": "Invalid Request" }` | `Q-006` |
