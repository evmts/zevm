# Context: cat2-004 — Add current_block_number field to Database

## Ticket Info
- **Ticket ID**: cat2-004
- **Category**: architecture
- **Goal**: Add a `current_block_number` field to the `Database` struct to track the canonical head block number, enabling `eth_blockNumber` RPC method implementation.

---

## Background

The `eth_blockNumber` JSON-RPC method returns the number of the most recent block (the canonical head). Currently, the `Database` struct only has `block_hashes` (a map of number→hash for the 256 most recent blocks), but no direct way to track the current head block number.

### Current Database Structure

**File**: `src/database/database.zig`
```zig
pub const Database = struct {
    state: state_manager.StateManager,
    accounts: @import("accounts.zig").Accounts,
    contracts: @import("contracts.zig").Contracts,
    block_hashes: @import("block_hashes.zig").BlockHashes,
    // MISSING: current_block_number field
};
```

**File**: `src/database/block_hashes.zig`
```zig
pub const BlockHashes = struct {
    hashes: std.AutoHashMapUnmanaged(u64, primitives.Hash.Hash) = .empty,
    // Stores mapping of block number -> hash for BLOCKHASH opcode (256 blocks)
};
```

The `block_hashes` map is only for the 256 most recent blocks and is designed for the `BLOCKHASH` opcode, not for tracking the canonical head.

---

## Requirements

### 1. Add `current_block_number: u64` to Database

The field should track the highest block number in the canonical chain.

### 2. Initialize to 0 (Genesis)

On fresh database initialization, `current_block_number` should be 0 (genesis block).

### 3. Update on Block Finalization

When a new block is built and finalized via `block_builder`, the `current_block_number` should be updated.

**File**: `src/block_builder.zig`
```zig
pub const BlockResult = struct {
    receipts: []primitives.Receipt.Receipt,
    total_gas_used: u64,
    block_number: u64,  // This is the new block number
    ...
};
```

### 4. Thread Safety Considerations

The `current_block_number` should be updated atomically with block hash storage to ensure consistency.

## Upstream Dependencies

### voltaire Blockchain (already has head tracking)

**File**: `../voltaire/packages/voltaire-zig/src/blockchain/Blockchain.zig`
```zig
pub const Blockchain = struct {
    block_store: BlockStore,
    fork_cache: ?*ForkBlockCache,

    /// Get current head block number (local canonical chain)
    pub fn getHeadBlockNumber(self: *Blockchain) ?u64 {
        return self.block_store.getHeadBlockNumber();
    }
};
```

**File**: `../voltaire/packages/voltaire-zig/src/blockchain/BlockStore.zig`
```zig
pub const BlockStore = struct {
    blocks: std.AutoHashMap(Hash.Hash, Block.Block),
    canonical_chain: std.AutoHashMap(u64, Hash.Hash),
    orphans: std.AutoHashMap(Hash.Hash, void),

    /// Get current head block number
    pub fn getHeadBlockNumber(self: *BlockStore) ?u64 {
        var max_number: ?u64 = null;
        var it = self.canonical_chain.keyIterator();
        while (it.next()) |number| {
            if (max_number == null or number.* > max_number.?) {
                max_number = number.*;
            }
        }
        return max_number;
    }
};
```

Voltaire's `Blockchain` already tracks the head by iterating the canonical chain. However, zevm's `Database` is a separate abstraction used for state management and needs its own head tracking for `eth_blockNumber`.

## Proposed Changes

### 1. Database struct addition

**File**: `src/database/database.zig`
```zig
pub const Database = struct {
    state: state_manager.StateManager,
    accounts: @import("accounts.zig").Accounts,
    contracts: @import("contracts.zig").Contracts,
    block_hashes: @import("block_hashes.zig").BlockHashes,
    current_block_number: u64,  // NEW: tracks canonical head

    pub fn init(allocator: std.mem.Allocator) !Database {
        return .{
            .state = try state_manager.StateManager.init(allocator, null),
            .accounts = @import("accounts.zig").Accounts.init(allocator),
            .contracts = @import("contracts.zig").Contracts.init(),
            .block_hashes = @import("block_hashes.zig").BlockHashes.init(),
            .current_block_number = 0,  // NEW: start at genesis
        };
    }
};
```

### 2. Block finalization update

The `current_block_number` should be updated when a block is built. This happens in coordination with the code that calls `block_builder.buildBlock()` and then stores the block hash.

Pattern from `BlockResult`:
```zig
const result = try block_builder.buildBlock(...);
// After successful block building:
db.current_block_number = result.block_number;
try db.block_hashes.put(allocator, result.block_number, block_hash);
```

### 3. eth_blockNumber RPC handler (future)

Once the field exists, the `eth_blockNumber` handler can be implemented:
```zig
pub fn blockNumber(db: *Database) u64 {
    return db.current_block_number;
}
```

## Zig Style Notes

- Do NOT create local type aliases like `const BlockNumber = u64`. Use `u64` directly.
- Do NOT store allocator in Database. Pass it explicitly to methods that need it.
- The `current_block_number` field is a primitive `u64`, so it requires no special deinit handling.
