## Normative API Contracts for the 9 Methods

Primary spec source: `execution-apis/src/eth/client.yaml`, `execution-apis/src/eth/state.yaml`, `execution-apis/src/eth/fee_market.yaml`, and schema files under `execution-apis/src/schemas/`.

| Method | Params | Result | Key spec files |
|---|---|---|---|
| `eth_chainId` | `[]` | `uint` (Quantity string) | `execution-apis/src/eth/client.yaml` |
| `eth_blockNumber` | `[]` | `uint` (Quantity string) | `execution-apis/src/eth/client.yaml` |
| `eth_getBalance` | `[address, BlockNumberOrTagOrHash]` | `uint` | `execution-apis/src/eth/state.yaml` |
| `eth_getCode` | `[address, BlockNumberOrTagOrHash]` | `bytes` | `execution-apis/src/eth/state.yaml` |
| `eth_getStorageAt` | `[address, bytesMax32, BlockNumberOrTagOrHash]` | `bytes` (32-byte value) | `execution-apis/src/eth/state.yaml` |
| `eth_getTransactionCount` | `[address, BlockNumberOrTagOrHash]` | `uint` (nonce) | `execution-apis/src/eth/state.yaml` |
| `eth_gasPrice` | `[]` | `uint` | `execution-apis/src/eth/fee_market.yaml` |
| `eth_coinbase` | `[]` | `address` | `execution-apis/src/eth/client.yaml` |
| `eth_accounts` | `[]` | `address[]` | `execution-apis/src/eth/client.yaml` |

Schema details:
- `uint`: `^0x(0|[1-9a-f][0-9a-f]*)$` (minimal hex quantity)
- `address`: `^0x[0-9a-fA-F]{40}$`
- `bytesMax32`: `^0x[0-9a-f]{0,64}$`
- `BlockNumberOrTagOrHash`: number, tag (`earliest|finalized|safe|latest|pending`), or hash

Findings:

Implication for cat2-006:
- If zevm uses these types directly, `eth_accounts` type correctness depends on upstream fix in voltaire.

## Handler Mapping for cat2-006

| Handler | Data source in ticket | Practical source call |
|---|---|---|
| `eth_chainId` | `NodeConfig` | `config.chain_id` |
| `eth_blockNumber` | block state | `db.current_block_number` (or equivalent canonical head source) |
| `eth_getBalance` | `StateManager.getBalance` | `db.state.getBalance(address)` |
| `eth_getCode` | `StateManager.getCode` | `db.state.getCode(address)` |
| `eth_getStorageAt` | `StateManager.getStorage` | `db.state.getStorage(address, slot)` |
| `eth_getTransactionCount` | `StateManager.getNonce` | `db.state.getNonce(address)` |
| `eth_gasPrice` | `NodeConfig.base_fee_per_gas` or 1 gwei | config value, else `1_000_000_000` |
| `eth_coinbase` | `NodeConfig.coinbase` | `config.coinbase` |
| `eth_accounts` | `NodeConfig.managed_accounts` | `config.managed_accounts` |

Formatting requirements from references:
- Quantities must be minimal `0x` hex.
- `eth_getCode` returns byte `Data` (`0x` for empty code).
- `eth_getStorageAt` result should be 32-byte left-padded hex.
- Addresses must be 20-byte `0x` hex.

## Key Risks / Open Questions Before Implementation

1. **`NodeConfig` dependency is not yet present in code**
- Ticket requires handlers to accept `*NodeConfig`.
- `src/node_config.zig` does not exist in current checkout.

2. **`eth_blockNumber` source in `Database` is not yet present**
- Ticket says read from block state.
- `Database` currently has no `current_block_number` field in implementation.

3. **voltaire `eth_accounts` result type mismatch**
- Upstream file still typed as `types.Quantity`.
- Must be corrected upstream (voltaire) to avoid wrong wire shape when typed serialization is used.

4. **Block selector semantics**
- Specs and references support block tags/number/hash.
- StateManager API in zevm integration layer currently reads latest in-memory state directly; historical block-state selection behavior needs explicit design if strict conformance is required.
