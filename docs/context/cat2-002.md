# Ticket cat2-002: Fix eth_getCode Result Type

## Ticket Summary

**Title:** Fix voltaire eth_getCode Result type: change from Quantity to a Bytes type

**Description:** In voltaire/packages/voltaire-zig/src/jsonrpc/eth/getCode/eth_getCode.zig, `Result.value` is typed as `types.Quantity`. The eth_getCode spec returns hex-encoded bytecode which is raw bytes, not a numeric quantity. Create or use an existing Bytes type (or []u8 with hex encoding) and update the Result struct. Quantity implies numeric semantics which is wrong for bytecode.

**Category:** spec-compliance

---

## Research Findings

### 1. The Problem

The current voltaire implementation incorrectly types `eth_getCode` result as `types.Quantity`:

```zig
// File: voltaire/packages/voltaire-zig/src/jsonrpc/eth/getCode/eth_getCode.zig (lines 43-45)
pub const Result = struct {
    /// hex encoded bytes
    value: types.Quantity,  // WRONG - should be bytes type
    ...
};
```

`types.Quantity` per EIP-1474 is:
- A hex-encoded number with **NO leading zeros** (except "0x0")
- Used for numeric values like balances, nonces, gas limits
- Pattern: `^0x(0|[1-9a-f][0-9a-f]*)$`

`eth_getCode` returns **bytecode**, which is:
- Arbitrary-length hex-encoded **raw bytes**
- Must preserve leading zeros (e.g., `0x001234` is valid)
- Pattern: `^0x[0-9a-f]*$`

### 2. Official Specification

From the Execution APIs OpenRPC spec:

**File:** `execution-apis/src/eth/state.yaml` (lines 84-108)

The `bytes` schema is defined as:
- Hex-encoded arbitrary-length byte string
- 0x-prefixed
- Even number of hex digits
- Empty string represented as "0x"

**Pattern:** `^0x[0-9a-f]*$`

**Reference:** `execution-apis/schema.graphqls` (lines 234-237)

## Implementation Plan

### Step 1: Create Bytes Type (in voltaire)

Create new file: `voltaire/packages/voltaire-zig/src/jsonrpc/types/Bytes.zig`

```zig
const std = @import("std");

/// BYTES per EIP-1474 (0x-prefixed hex for arbitrary-length byte strings).
/// Unlike Quantity, Bytes preserves leading zeros and is used for raw data
/// like contract bytecode, storage values, and transaction input data.
pub const Bytes = struct {
    /// Owned byte array - caller must free after use
    data: []u8,

    pub fn jsonStringify(self: Bytes, jws: *std.json.Stringify) !void {
        // Use Hex.bytesToHex or std.fmt implementation
        const hex = try std.fmt.allocPrint(jws.allocator, "0x{}", .{std.fmt.fmtSliceHexLower(self.data)});
        defer jws.allocator.free(hex);
        try jws.write(hex);
    }

    pub fn jsonParseFromValue(allocator: std.mem.Allocator, source: std.json.Value, _: std.json.ParseOptions) !Bytes {
        switch (source) {
            .string => |s| {
                if (s.len < 2 or s[0] != '0' or (s[1] != 'x' and s[1] != 'X')) {
                    return error.InvalidBytes;
                }
                const hex_digits = s[2..];
                if (hex_digits.len % 2 != 0) {
                    return error.InvalidBytes; // Odd number of hex digits
                }
                const bytes = try allocator.alloc(u8, hex_digits.len / 2);
                errdefer allocator.free(bytes);
                _ = std.fmt.hexToBytes(bytes, hex_digits) catch return error.InvalidBytes;
                return Bytes{ .data = bytes };
            },
            else => return error.InvalidBytes,
        }
    }

    pub fn deinit(self: Bytes, allocator: std.mem.Allocator) void {
        allocator.free(self.data);
    }
};
```

```zig
            .value = try std.json.innerParseFromValue(types.Bytes, allocator, source, options),

    }
};
```

## Summary

The fix requires:

1. **Creating** a new `Bytes` JSON-RPC type in voltaire (variable-length byte array with proper hex serialization)
2. **Updating** `eth_getCode.Result.value` from `types.Quantity` to `types.Bytes`
3. **Also fixing** `eth_getStorageAt` which has the same issue

This ensures spec compliance: `eth_getCode` returns arbitrary-length hex-encoded bytecode, not a numeric quantity.
