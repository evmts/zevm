# Plan: cat2-001 — Fix voltaire eth_accounts Result type

## Overview

The `eth_accounts` JSON-RPC method in voltaire has its `Result.value` typed as `types.Quantity` (a hex-encoded integer). Per the Ethereum execution-APIs spec (`execution-apis/src/eth/client.yaml`), the result must be a JSON array of 20-byte hex-encoded addresses (`[]types.Address`).

This is a single-file fix in voltaire (upstream dependency we own):
`../voltaire/packages/voltaire-zig/src/jsonrpc/eth/accounts/eth_accounts.zig`

No downstream consumers in zevm or guillotine-mini reference `eth_accounts.Result` directly yet, so this is a safe, backwards-compatible-at-the-wire-level correction.

## Approach

Change `Result.value` from `types.Quantity` to `[]types.Address`, then update `jsonStringify` to write a JSON array of addresses, and `jsonParseFromValue` to parse a JSON array source into an allocated `[]types.Address` slice.

### Step 3: Update `jsonStringify`

Replace the `jsonStringify` body to iterate over the address slice and write each element inside a JSON array:

```zig
pub fn jsonStringify(self: Result, jws: *std.json.Stringify) !void {
    try jws.beginArray();
    for (self.value) |addr| {
        try jws.write(addr);
    }
    try jws.endArray();
}
```

This follows the same `beginArray`/`write`/`endArray` pattern used by Params serialization throughout voltaire (e.g., `debug_getRawTransaction.zig` Params, `eth_getLogs.zig` Params).

### Step 4: Update `jsonParseFromValue`

Replace the `jsonParseFromValue` body to validate the source is a JSON array, allocate a slice, and parse each element as `types.Address`:

```zig
pub fn jsonParseFromValue(allocator: std.mem.Allocator, source: std.json.Value, options: std.json.ParseOptions) !Result {
    if (source != .array) return error.UnexpectedToken;
    const items = source.array.items;
    const addrs = try allocator.alloc(types.Address, items.len);
    for (items, 0..) |item, i| {
        addrs[i] = try std.json.innerParseFromValue(types.Address, allocator, item, options);
    }
    return Result{ .value = addrs };
}
```

## Risks and Mitigations

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `std.json.Stringify.beginArray`/`endArray` API mismatch | Low | Already used extensively in voltaire Params serialization (confirmed in 50+ files) |
| `types.Address.jsonStringify` doesn't work inside `jws.write()` | Low | `eth_coinbase` already does `jws.write(self.value)` with a single `types.Address` — same pattern, just in a loop |
| Breaking downstream zevm consumers | None | No zevm `.zig` file references `eth_accounts.Result` yet |
| Breaking `methods.zig` union | None | The union field `result: eth_accounts.Result` is structural — changing the inner type is transparent |
| Memory leak from allocated slice | Low | Document that callers must free `result.value` via the allocator. This matches the existing pattern for all `jsonParseFromValue` methods that allocate (standard Zig JSON parsing convention) |
| `[]const types.Address` vs `[]types.Address` | Low | Use `[]const types.Address` to match Zig slice conventions for read-only data returned from parsing |
